# 迁移分析

## 核心结论

把 `claude-code` 改造成更适配 Codex 的产品，难点不在协议字段，而在两边对代理系统的默认假设完全不同。

`Claude Code` 更像一套围绕 Claude 行为打磨出来的执行外壳：

- 依赖长系统提示和提示缓存切块。
- 依赖 `tool_use / tool_result` 的回合结构。
- 依赖 Claude 的思考开关、测试特性开关、`fast mode`、`1M` 上下文和 `advisor` 工具。
- 依赖一批会影响模型行为的附加提示内容，比如 agent 列表、todo 提醒、task 提醒。

`Codex` 更像一套围绕结构化代理工作流打磨出来的执行外壳：

- 显式定义用户输入、推理、工具调用、工具结果、最终输出阶段。
- 显式定义模型能力，如推理档位、输入模态、并行工具、输出详细度。
- 显式定义权限申请、沙箱边界和说明文字。
- 显式定义子代理协作、等待时机和收尾动作。

所以，真正需要迁移的是“内部代理架构”，不是“先把一个客户端换成另一个客户端”。

这里要再说清楚一点：很多能力最后都会体现在请求和消息里，但不能把 coding agent 简化成“把规则写成消息发给模型就行”。

真正要区分的是三类东西：

- 只需提示词或描述适配的：例如部分工具描述、权限文案、系统提示词里的措辞和顺序。
- 需要改消息生产规则的：例如用户输入怎么归一化、工具结果怎么回灌、阶段化输出怎么分层、哪些上下文该进消息、哪些不该进。
- 必须由本地执行链实现的：例如启动链、REPL、`QueryEngine`、`query`、工具执行、权限记录、结果回灌、TUI 渲染、状态持久化。这些不能只靠模型回复替代。

## Claude Code 里真正绑死 Claude 的部分

### 模型接入层

- 查询入口直接走 `queryModelWithStreaming`。
- 客户端直接实例化 Anthropic 及其几个云平台变体。

这意味着它现在不是“模型无关壳层”，而是“以 Claude 为中心的产品实现”。

### 请求构造层

- 请求体直接围绕 Anthropic Beta Messages 组装。
- `betas`、`thinking`、`context_management`、`output_config`、`speed`、`advisor_20260301` 都是公共路径的一部分。

这已经不是“字段名不同”，而是整条请求路径都带着 Claude 的假设。

### 流式事件层

- 上层直接理解 `message_start`、`content_block_delta`、`tool_use`、`server_tool_use`、`thinking_delta`、`signature_delta`。

只要这层不拆，任何新模型都得先伪装成 Claude 的事件流，后续维护会越来越别扭。

### 模型调教层

- 思考开关和自适应思考规则写死在 Claude 模型族上。
- `/model`、`fast mode`、`1M`、额外计费说明都写死在 Claude 产品线上。

这类逻辑如果继续留在公共层，后面所有新模型接入都会被旧命名和旧假设拖住。

### 工具与会话治理层

- agent 列表、todo 提醒、task 提醒、viewer 行为、权限成功判定都围绕 `tool_use / tool_result`。
- 缓存和系统提示则通过 `cache_control`、缓存断点、header latch 来维持稳定。

这些逻辑都很有价值，但它们现在附着在 Claude 的回合结构和本地执行链上，需要拆出来重新挂接，而不是只改几段提示词。

## Codex 值得直接借鉴的设计

### 统一中间表示

Codex 先定义中间对象，再去适配传输层。这样模型、工具、界面、存储不会直接缠死在某一家供应商的协议上，也不会把本地执行链误写成一堆松散消息。

### 模型能力表

Codex 的模型能力不是散落在 `if model.includes(...)` 里，而是模型元数据的一部分。这个设计更适合长期维护，也更适合以后做多模型。

### 权限与沙箱是正式输入

Codex 把 `sandbox_permissions`、`justification`、`prefix_rule` 直接做成工具输入，再由界面去表达“允许一次、允许本会话、永久允许”。这比事后弹一个确认框更稳。

### 子代理协作是一等能力

Codex 把派子代理、发消息、等待结果、关闭代理都当成正式工具和正式流程，而不是隐藏在提示词背后。

### 阶段化输出

Codex 把中间说明和最终答案拆成不同阶段。对于终端代理来说，这会直接影响进度展示、工具回合结束时机和最终收口方式。

## 我们真正要做的事

第一，不要先换客户端。

第二，不要先做 `messages -> responses` 的浅层翻译。

第三，不要把“消息层”和“本地执行链”拆成互不相干的两件事。消息当然重要，但消息是本地执行链产物的一部分，不是全部。

第四，不要先改模型名和界面文案。

正确顺序应该是：

1. 抽内部中间层，统一表示用户输入、助手中间输出、最终输出、推理、工具调用、工具结果、权限请求、压缩事件。
2. 保住并接管 Claude Code 现有的本地执行链，至少包括启动链、REPL、`QueryEngine`、`query`、工具执行、权限、结果回灌、TUI 渲染，不把它们模糊写成“外围系统”。
3. 抽模型能力表，把 Claude 产品词从公共层移出去。
4. 重写工具和权限循环，让上层不再直接依赖 `tool_use / tool_result`。
5. 再接入 Codex 后端，并逐步重做模型选择、文案和更多增强能力。

## 对第三方分析仓库的判断

`sanbuphy/claude-code-source-code` 对我们有帮助，但定位只能是二级正式参考。

它真正有价值的地方是：

- 把 CLI 启动层、主循环、工具执行、权限链路、压缩策略、远端桥接、任务式多代理拆得更清楚。
- 能帮助我们更快列出“Claude Code 为什么好用”的能力清单。

它的边界也很明确：

- 它不是 Anthropic 原始单仓库，而是基于已发布包还原出的研究仓库。
- 它缺一批在构建期被裁掉的模块，尤其是一些更深层的压缩和协作能力。
- 它混有作者自己的解释和补丁脚本，不能直接替代原始源码结论。

所以，我们的使用规则是：

- 用它做结构导航、检查清单和设计启发。
- 用本仓库导入的 `claude-code` 快照做正式基线。
- 用官方文档固定公开产品事实和发布时间。
