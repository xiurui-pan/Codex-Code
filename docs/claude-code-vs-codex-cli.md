# 为什么 Claude Code 比 Codex CLI 更好用

## 直接回答

是，在很多核心使用感上，`Claude Code` 目前确实比 `Codex CLI` 更好用，而且原因不只是界面。

这里说的“更好用”，不是单指看起来更顺手，而是指它在真实编码工作里的整套执行外壳更成熟：更会管权限、更会维持长会话、更会组织任务、更会处理远端会话，也更会把工具、钩子和交互节奏串成一个稳定产品。

## Claude Code 更强的地方

### 1. 权限流程更完整

Claude Code 不是“遇到命令就弹个确认框”。它把工作区信任、危险模式确认、MCP 审批、远端权限展示、权限请求钩子和持久化规则做成了一整条流程。

这会直接影响日常可用性：

- 用户不容易被一堆零散确认打断。
- 危险操作的边界更清楚。
- 本地与远端会话的许可状态更一致。
- 自动模式也不会轻易演变成“默认全放行”。

### 2. 长会话治理更成熟

Claude Code 对上下文压缩、提醒拼装、工具描述稳定化、压缩前后衔接这些问题投入更深。

这类能力平时不显眼，但会直接决定一个终端代理在长时间编码后是不是还好用：

- 会不会越聊越乱。
- 会不会因为工具输出太长而失控。
- 压缩后还能不能继续做事。
- 提醒和附加说明会不会干扰模型稳定性。

### 3. 任务拆分和多代理能力更产品化

Claude Code 的多代理不是零散调用，而更接近一套任务系统。任务有启动、通知、输出、恢复和隔离边界，远端与本地也能接起来。

这会让“让另一个代理去调查、验证、改动、回报”变成稳定能力，而不是一次性技巧。

### 4. 工具与钩子生态更完整

Claude Code 的价值不只在内置工具，更在它围绕工具做了一层更完整的工作台：

- 工具列表和暴露策略
- MCP 接入
- 用户钩子
- 权限请求钩子
- 压缩前后钩子
- 工作树创建钩子

这套外壳让它更像一个可长期扩展的终端工作环境，而不是一个单纯的模型命令行。

### 5. 交互节奏更成熟

Claude Code 在很多细节上更像“已经打磨过的终端产品”：

- 什么时候给用户进度更新
- 什么时候进入工具回合
- 什么时候请求许可
- 什么时候压缩
- 什么时候结束本轮回答

这些细节决定的是使用时的顺滑程度，不是界面颜色或排版。

### 6. 远端和后台能力更成体系

Claude Code 对远端会话、桥接、状态同步、权限请求转发、会话恢复这些能力投入很多。即使用户暂时不用远端，这套能力也会反过来提升本地实现的清晰度，因为它逼着系统把状态和事件定义得更完整。

## Codex CLI 更强的地方

Codex CLI 也不是单纯“不好用”。它真正更强的是底层建模：

- 更适配 Codex 模型本身
- 更强调结构化输入和结构化工具
- 更明确地区分中间输出和最终输出
- 更正式地表达权限申请和沙箱边界
- 更明确地维护模型能力表

也就是说，Codex CLI 的强项更多在“底层模型适配”和“结构定义”，而 Claude Code 的强项更多在“产品外壳”和“交互编排”。

## 对 Codex Code 的启发

这也是 `Codex Code` 要做的事情：

- 保留 Claude Code 这套更成熟的执行外壳。
- 引入 Codex 这套更适合 Codex 模型的结构化适配方式。
- 不把问题误解成“换个接口”或“换个界面主题”。

如果只换接口，不换内部结构，最后得到的通常会是“能调用 Codex，但行为还是 Claude Code 老习惯”的半成品。

如果只换界面，不换模型适配和工具循环，最后得到的只是“长得像 Claude Code 的 Codex 命令行”。

真正有价值的目标，是把两边最强的部分接起来：

- 用 Claude Code 的产品外壳承接用户操作和工具流程。
- 用 Codex 的结构化模型适配承接底层回合、工具调用、权限和阶段化输出。

这也是本仓库后续开发的总方向。
